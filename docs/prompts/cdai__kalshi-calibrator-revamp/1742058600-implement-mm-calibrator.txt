# Plan: Market-Making Loss Calibrator (Issue #40)

## Context

The current calibration pipeline is: `kalshi fetch` → normalize probabilities to CSV → `calibrate` fits goose values using Beta-CI convergence. The normalization pipeline (backfill, 5-pass rescaling, monotonicity enforcement) is fragile and lossy. Issue #40 proposes replacing it with a market-making loss function: measure how much money our model would make trading against Kalshi's orderbook. An efficient model makes $0.

User requirements: HTTPS-only (no WS) for V1, use cache with TTL, fetch **top N orderbook levels** (not just NBBO) to penalize passing through spread, build modularly for future complexity.

## Implementation Steps

### Phase 1: Kalshi Orderbook Types & Fetching

**File: `crates/kalshi/src/types.rs`** — Add types:
- `OrderbookLevel { price: u32, quantity: u32 }` — single price level (cents, contracts)
- `Orderbook { ticker: String, yes_bids: Vec<OrderbookLevel>, yes_asks: Vec<OrderbookLevel> }`
- `OrderbookResponse` / `OrderbookResponseInner` — raw API deserialization (`{"orderbook": {"yes": [[p,q],...], "no": [[p,q],...]}}`)
- `CachedOrderbooks { fetched_at, event_ticker, round, orderbooks: Vec<Orderbook> }`
- `TeamOrderbook { team: String, round: usize, ticker: String, orderbook: Orderbook }`

**File: `crates/kalshi/src/rest.rs`** — Add methods:
- `get_orderbook(ticker: &str, depth: usize) -> Result<Orderbook>` — `GET /trade-api/v2/markets/{ticker}/orderbook?depth={depth}`, parse YES bids/asks from response (NO bids at price X → YES asks at 100-X)
- `get_round_orderbooks(market_def, markets, depth, sleep_ms) -> Result<Vec<Orderbook>>` — batch fetch for all tickers in a round
- `load_orderbook_cache(market, ttl)` / `save_orderbook_cache(market, orderbooks)` — same pattern as existing `load_cache`/`save_cache`

**File: `crates/kalshi/src/lib.rs`** — Add `pub mod orderbook;`

### Phase 2: Edge Computation Module

**New file: `crates/kalshi/src/orderbook.rs`** — Market-making loss computation:

```rust
// Core types
```
