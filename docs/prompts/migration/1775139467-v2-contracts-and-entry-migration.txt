Summary
We need a one-time clean-slate migration from the current legacy UI/off-chain bracket encoding to the contract-correct ByteBracket encoding.

This is now the operational path forward for the 2026 testnet contest.

The key decision is:

after cutover, contract-correct bytes8 becomes the only runtime bracket format everywhere
legacy 63-bit reversal exists only in one-time migration tooling
This intentionally avoids carrying legacy compatibility shims into next year's mainnet launch.

Related: #251

Constraints / decisions
Submissions are already closed, so we do not need to preserve the normal submission flow during migration.
The site can go down for a one-night cutover.
Main contract entries were submitted through the UI, so existing on-chain brackets should be treated as legacy-encoded and must be migrated.
Groups are critical and must be migrated.
Mirrors are not critical, are owner-controlled, and do not handle funds; they do not block the main migration.
This is testnet money, so redeploying contracts and re-funding them is acceptable.
We want the post-cutover system to be battle-tested for next year's mainnet, not a temporary dual-encoding architecture.
Contract changes should be minimized: prefer thin V2 contracts that inherit the current V1 logic and only add migration/admin capabilities.
Target state
After cutover:

the UI/client encodes and decodes brackets in contract-correct order
the main contract stores contract-correct brackets
BracketGroups points at the new main contract
Redis/indexer/server store contract-correct hex under the normal runtime keys
Rust scoring/simulation consumes contract-correct hex natively
the only legacy code is in one-time migration scripts
